# CLAUDE.md — もみじストアOS 標準ルール

## プロジェクト概要

「もみじストアOS」の開発環境。
EC運営の自動化・在庫管理・楽天/Amazon/メルカリ連携・AIエージェントによる業務効率化を目的とする。

詳細な開発ルールは `MomijiStore_OS/04_Manual/DeveloperGuide/` に分離予定。

---

## 作業範囲

**作業対象:** `Desktop/Claude Code` 配下のみ

**ワークスペース外は変更禁止。**
外部ファイル（~/Documents、~/Downloads、外付けSSD、NAS、iCloud Drive 等）への読み書きは一切行わない。

---

## 安全ルール

| ルール | 詳細 |
|--------|------|
| 本番ファイル直接編集禁止 | `_本番用` ファイルは承認なく編集しない |
| 作業前にバックアップ取得 | 既存ファイルの変更前に必ずバックアップを作成する |
| 削除・移動・リネーム承認制 | 実行前にユーザーへ確認・承認を得る |
| 古いバックアップ自動削除禁止 | 不要なバックアップは削除候補として提示するのみ |
| 大規模変更は事前説明 | 変更内容・影響範囲を先に提示してから実施する |

---

## 実装ルール

- 勝手な仕様変更禁止。改善案は**提案のみ**とし、実装は承認後に行う。
- 安全性・保守性・拡張性を最優先する。
- コードは可読性重視。コメントは WHY を中心に記述する。
- ハードコーディング禁止。ファイル名は英語で統一。

---

## 作業手順

1. 変更内容・影響範囲をユーザーへ提示
2. 承認を得る
3. バックアップ取得
4. 実装
5. 動作確認・報告

---

## 作業完了時の標準フロー

すべての作業完了時は、以下の手順を必ず実施する。

1. 作業前バックアップが存在することを確認する
2. 実装内容のテストを実施する
3. テスト結果（PASS / FAIL）を報告する
4. PASS の場合のみ GitHub へコミット・Push を行う
5. コミットメッセージは変更内容が分かる形式で作成する
6. 作業内容・変更ファイル・テスト結果・GitHub 反映結果を最後に報告する
7. 次回推奨作業を3件以内で提案する

### GitHub 運用ルール

- コミット前に `git status` を確認する
- Push 後は現在のブランチ名・コミットハッシュ・GitHub 反映結果を報告する
- リリース時は `CHANGELOG.md` を更新する
- 正式リリース時は GitHub Release の作成を提案する

### エラー時

テストが FAIL の場合は GitHub へ Push を行わず、以下を報告して承認を待つ。

- 原因
- 影響範囲
- 修正案

---

## Git 運用

- 変更前にコミット（バックアップ兼用）
- 機能単位でコミット
- `main` への直接大規模変更禁止
